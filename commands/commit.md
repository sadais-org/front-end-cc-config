# Git Commit 命令（智能提交）

## 命令说明

这是一个增强版的 git commit 命令，在提交前可以：
1. ❓ **可选的代码质量检查**（code-simplifier，询问用户是否需要）
2. 🔄 **支持循环优化**（直到用户满意）
3. 📝 **生成规范的提交消息**（约定式提交 + emoji）
4. 🚀 **自动推送到远程**

---

## 工作流程总览

```
/commit
  ↓
【准备阶段】
  ├─ 1. 收集 git 变更信息
  └─ 2. 询问：是否需要代码质量检查？
        ├─ 是 → 进入第一阶段
        └─ 否 → 直接跳到第二阶段
  ↓
【第一阶段：代码质量检查】（可选，可循环）
  ├─ 3. code-simplifier 分析代码
  ├─ 4. 有优化建议？
  │     ├─ 是 → 询问用户 → 处理 → 回到步骤 3
  │     └─ 否 → 进入第二阶段
  ↓
【第二阶段：Git 提交流程】
  ├─ 5. 分析变更内容
  ├─ 6. 拆分提交（按类型）
  ├─ 7. 生成提交消息（约定式提交）
  ├─ 8. 执行 git add + commit
  └─ 9. 推送到远程
```

---

## 准备阶段

### 步骤 1：收集 Git 变更信息

使用 Bash tool 收集以下信息：

```bash
# 1. 查看当前状态
git status

# 2. 查看所有变更（已暂存 + 未暂存）
git diff HEAD

# 3. 获取当前分支
git branch --show-current

# 4. 查看最近的提交历史
git log --oneline -10
```

**输出：**
- 修改的文件列表
- 具体的代码变更
- 当前分支名称
- 提交历史（用于参考提交风格）

**特殊情况处理：**
- 如果没有任何变更（git status 显示 clean），提示用户并结束

---

### 步骤 2：询问是否需要代码质量检查

使用 AskUserQuestion 工具询问：

```json
{
  "questions": [{
    "question": "是否需要在提交前进行代码质量检查？",
    "header": "质量检查",
    "multiSelect": false,
    "options": [
      {
        "label": "跳过检查，直接提交",
        "description": "快速提交，不进行代码质量分析（推荐用于紧急修复）"
      },
      {
        "label": "进行质量检查",
        "description": "使用 code-simplifier 分析代码，确保质量达标"
      }
    ]
  }]
}
```

**根据用户选择：**
- **选择"跳过检查，直接提交"** → 直接跳到【第二阶段：Git 提交流程】
- **选择"进行质量检查"** → 继续【第一阶段：代码质量检查与优化】

---

## 第一阶段：代码质量检查与优化

**前置条件：** 用户在步骤 2 选择了"进行质量检查"

**目标：** 确保提交的代码符合质量标准，无明显可优化点

---

### 步骤 3：调用 code-simplifier 分析代码

**目的：** 仅分析，不修改代码，给出优化建议

使用 Task tool：
- `subagent_type`: "code-simplifier:code-simplifier"
- `description`: "分析待提交代码"
- `prompt`:
  ```
  分析以下待提交的代码，给出优化建议（不要修改代码）：

  待分析的文件（从 git diff 获取）：
  [列出所有修改的文件和主要变更]

  分析重点：
  - 复杂逻辑是否需要简化
  - 是否存在重复代码
  - 代码结构是否可以优化
  - 是否符合项目规范（CLAUDE.md）
  - TypeScript 类型是否安全

  只给出分析结果和优化建议，不要修改代码。
  如果代码质量良好，无需优化，请明确说明。
  ```

---

### 步骤 4：判断是否有优化建议

根据 code-simplifier 的分析结果：

**4.1 如果没有优化建议（代码质量良好）：**
- 向用户说明："✅ 代码质量检查通过，无需优化"
- **直接跳到【第二阶段：Git 提交流程】**

**4.2 如果有优化建议：**
- 继续步骤 5

---

### 步骤 5：展示分析结果

向用户清晰展示：
- 📊 code-simplifier 的分析结果
- 📝 具体的优化建议列表
- 📂 影响的文件和代码位置
- ⚠️ 优化的优先级（重要/一般）

---

### 步骤 6：询问用户决策

使用 AskUserQuestion 工具：

```json
{
  "questions": [{
    "question": "发现以下代码可以优化，是否要处理？",
    "header": "代码优化",
    "multiSelect": false,
    "options": [
      {
        "label": "处理优化建议",
        "description": "执行优化后会再次检查代码质量，确保达标"
      },
      {
        "label": "跳过，直接提交",
        "description": "保持当前代码，进入提交流程"
      }
    ]
  }]
}
```

---

### 步骤 7：根据用户选择执行

#### 7.1 如果用户选择"处理优化建议"：

1. **调用 code-simplifier 执行优化**（实际修改文件）

   使用 Task tool：
   - `subagent_type`: "code-simplifier:code-simplifier"
   - `description`: "执行代码优化"
   - `prompt`:
     ```
     根据之前的分析结果，执行以下优化：
     [列出具体的优化项]

     要求：
     - 保持所有功能不变
     - 遵循项目规范（CLAUDE.md）
     - 确保 TypeScript 类型安全
     - 优化后的代码更易读、易维护
     ```

2. **优化完成后，回到步骤 3**（重新分析优化后的代码）
   - 形成检查循环，确保优化到位
   - 循环直到没有更多优化建议或用户选择"跳过"

#### 7.2 如果用户选择"跳过，直接提交"：

- **进入【第二阶段：Git 提交流程】**

---

## 第二阶段：Git 提交流程

**前置条件：** 代码质量检查已完成（通过或用户选择跳过）

---

### 约定式提交规范（Conventional Commits）

**格式：** `<emoji> <type>: <subject>`

#### 提交类型和 Emoji 对照表

| Emoji | Type | 说明 | 示例 |
|-------|------|------|------|
| ✨ `:sparkles:` | feat | 新增功能 | `:sparkles: feat: 添加用户登录功能` |
| 🐛 `:bug:` | fix | 修复 bug | `:bug: fix: 修复登录失败的问题` |
| 📝 `:memo:` | docs | 更新文档 | `:memo: docs: 更新 API 文档` |
| 💄 `:lipstick:` | style | 样式调整（UI） | `:lipstick: style: 优化按钮样式` |
| ♻️ `:recycle:` | refactor | 重构代码 | `:recycle: refactor: 优化用户服务层` |
| ⚡ `:zap:` | perf | 性能优化 | `:zap: perf: 优化列表渲染性能` |
| ✅ `:white_check_mark:` | test | 添加测试 | `:white_check_mark: test: 添加登录单元测试` |
| 🔧 `:wrench:` | chore | 构建/工具变动 | `:wrench: chore: 更新依赖版本` |
| ⏪ `:rewind:` | revert | 回退版本 | `:rewind: revert: 回退到 v1.0.0` |
| 🔒 `:lock:` | security | 修复安全问题 | `:lock: security: 修复 XSS 漏洞` |
| 🚀 `:rocket:` | deploy | 部署 | `:rocket: deploy: 部署到生产环境` |
| 📦 `:package:` | build | 构建系统 | `:package: build: 更新 webpack 配置` |
| 🎨 `:art:` | format | 格式调整（代码） | `:art: format: 格式化代码` |
| 🔥 `:fire:` | remove | 删除代码/文件 | `:fire: remove: 删除废弃的 API` |

---

### 步骤 8：分析变更内容

基于 git diff 分析：
1. **变更性质**：新功能 / bug 修复 / 重构 / 样式调整...
2. **影响范围**：涉及哪些模块和文件
3. **变更目的**：为什么要做这个改动

**输出：** 对变更的清晰理解，为拆分提交做准备

---

### 步骤 9：拆分提交（按类型）

将所有变更按**提交类型**拆分成多个逻辑独立的提交：

**拆分原则：**
- ✅ 每个提交只做一件事（单一职责）
- ✅ 同一类型的变更放在一个提交
- ✅ feat 和 fix 不要混在一起
- ✅ 重构和新功能分开提交
- ✅ 子模块变更单独提交

**示例：**

假设有以下变更：
- 新增了用户头像上传功能（feat）
- 修复了登录按钮样式问题（style）
- 更新了 README 文档（docs）

应拆分为 3 个提交：
1. `:sparkles: feat: 添加用户头像上传功能`
2. `:lipstick: style: 修复登录按钮样式问题`
3. `:memo: docs: 更新 README 使用说明`

---

### 步骤 10：生成提交消息

对每个提交，生成 **3 个候选消息**：

**生成流程：**
1. 确定提交类型（参考约定式提交规范）
2. 确定 emoji 前缀
3. 用一句话概括变更内容
4. 生成 3 个候选，从不同角度描述

### 步骤 11：选择最佳提交消息

从 3 个候选中选择最合适的，并说明理由：

**选择依据：**
1. **参考最近的提交历史**（git log -10）
   - 保持团队风格一致
2. **准确性优先**
   - 准确描述变更内容
3. **简洁性**
   - 避免冗余词汇

**输出格式：**
```
✅ 最终选择: :sparkles: feat: 添加项目信息管理模块

选择理由:
- 简洁明了，一句话说清楚
- 符合团队提交风格（参考最近 10 条提交）
- 准确描述了核心功能
```

---

### 步骤 12：执行提交

对每个提交执行以下操作：

```bash
# 1. 暂存相关文件
git add <file1> <file2> ...

# 2. 创建提交（使用选定的提交消息）
git commit -m "<emoji> <type>: <subject>"

# 3. 验证提交成功
git log -1
```

**注意：**
- 使用 `git add` 精确暂存文件，避免 `git add .`
- 确保提交消息中不包含 Claude 共同作者页脚

---

### 步骤 13：推送到远程

所有提交完成后，推送到远程仓库：

```bash
git push
```

**如果推送失败：**
- 检查是否需要先 pull
- 检查是否有权限问题
- 向用户报告具体错误

---

### 步骤 14：处理多个提交

如果步骤 9 拆分出多个提交：
- **重复步骤 10-12**
- 逐个处理每个提交
- 最后统一推送（步骤 13）

---

## 约束条件

1. **不添加共同作者**
   - 不要在提交中添加 `Co-Authored-By: Claude` 页脚

2. **代码质量检查（可选）**
   - code-simplifier 检查是可选的，由用户在步骤 2 决定是否启用
   - 如果启用，优化循环必须由用户决定何时退出
   - 用户可以随时选择"跳过"直接进入提交流程

3. **提交规范强制**
   - 所有提交必须遵循约定式提交格式
   - 必须包含 emoji 前缀
   - 提交消息必须是中文

4. **单一职责**
   - 每个提交只做一件事
   - 不同类型的变更必须拆分成多个提交

5. **子模块处理**
   - 项目和子模块的变更必须分开提交
   - 先提交子模块，再提交主项目

6. **用户修改内容保护**
   - 在代码质量检查和优化阶段，禁止重置或覆盖用户已有的文件修改
   - code-simplifier 执行优化时，必须基于用户当前的工作区内容进行，不能还原到 git HEAD 版本
   - 若优化工具意外重置了用户的修改，必须立即中止流程并告知用户

---

**命令结束**
