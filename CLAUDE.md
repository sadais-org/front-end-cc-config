## 基础响应规范（强制执行）
- **全程中文回复**：所有面向用户的输出必须使用中文。
- **禁止技术内容切换英文**：无论任务复杂度如何、工具调用链多长，分析、说明、标题等用户可见内容都不得因技术性质切换为英文。
- **最新文档优先**：始终使用 Context7 MCP 获取最新的文档。
- **GitHub 操作使用 gh**：对于 GitHub 相关操作，必须使用 gh 命令行工具，而不是 WebFetch。

## 工作规范（强制执行）
- **代码前先确认方案**：在编写任何代码之前，必须先描述方案并等待用户批准。
- **需求不明确先澄清**：如果需求不明确，必须先提出澄清问题，并使用交互式方式向用户确认。
- **无确认不改文件**：用户未明确回复"确认"、"可以"、"ok"等同意信号时，禁止执行任何文件修改操作；工具加载消息（如"Tool loaded."）不属于确认信号。
- **超过 3 个文件先拆分**：如果一项任务需要修改超过 3 个文件，必须先停下来，将其分解成更小的任务。
- **修复前先读依赖**：分析或修复问题时，必须先梳理文件依赖树，通读全部相关文件后再处理。
- **交互确认必须用工具**：当流程中需要用户做出选择或确认时（如是否执行某操作、确认提交计划、选择处理方式等），必须调用 `AskUserQuestion` 工具提供选项让用户点选，禁止用纯文字提问代替，也不得仅在输出中文字提问后直接继续执行。

## 用户决策遵循规范（强制执行）
- **用户选择最高优先级**：用户通过 AskUserQuestion 做出的选择等同于直接指令，必须严格执行。
- **禁止改写用户选择**：收到用户选择后，禁止基于上下文、模型判断、已有计划或"更合理方案"进行改写、替换、合并、折中或扩展。
- **自定义输入原样使用**：用户自定义输入必须原样理解、原样使用；如需改写、格式转换或补全，必须再次用 AskUserQuestion 确认。
- **仅真实冲突可暂停**：只有在用户选择存在安全风险、破坏性操作、权限风险，或与系统/开发者级约束明确冲突时，才允许暂停并再次确认。
- **禁止伪造冲突**：不得把普通实现差异、模型偏好或流程习惯当作冲突来覆盖用户选择。
- **歧义必须确认**：如果无法唯一理解用户选择，必须再次用 AskUserQuestion 确认，禁止自行推断。

## 改动验证规范（强制执行）
- **每一步编辑后必须回读验证**：使用 Edit 工具执行修改后，必须用 Read 或 Grep 回读文件内容，确认改动是否真正写入磁盘，禁止在未验证的情况下声称操作成功
- **全部改动完成后必须全量扫描验证**：所有文件改动结束后，必须对涉及的每个文件逐项回读检查，确认每一项改动都实际生效，禁止基于"我执行了编辑操作"的假设就声称全部完成
- **跨会话恢复时必须做完整性校验**：从上一个 session 恢复时，禁止直接基于 summary 记录继续推进，必须逐文件回读验证历史改动是否真实持久化到磁盘，发现遗漏立即补全后再继续
- **区分"执行了操作"与"改动已生效"**：工具调用返回空结果不代表操作成功，只有回读确认文件内容符合预期才能判定成功。禁止在未确认生效的情况下输出"已完成"或标记任务为 completed

## 脚本执行规范（强制执行）
- **命令目录必须正确**：执行脚本命令时，必须时刻确认当前所处上下文目录，禁止在错误的上下文目录执行命令。

## 搜索策略规范（强制执行）
- **优先使用 rg**：搜索文件内容时必须优先使用 `rg`（ripgrep），若 rg 不可用或执行失败，则回退到 `grep`。
- **向上搜索到当前文件系统根**：在当前项目目录中搜索不到目标文件或符号时，必须逐级向上扩展搜索范围，直到当前路径所在的文件系统根目录（如 macOS/Linux 的 `/`，Windows 的当前盘符根目录如 `C:\`）。
- **跨盘搜索需确认**：在 Windows 等多根文件系统中，搜索到当前盘符根目录后仍未找到目标时，必须向用户确认是否需要搜索其它盘符或指定目录，禁止自行跨盘全量搜索。
- **搜索失败必须澄清**：若搜索到当前文件系统根后仍未找到目标，必须向用户澄清，禁止自行假设目标位置或跳过搜索。

## 清理规范（强制执行）
- **完成后删除临时文件**：任务完成后，必须删除执行过程中创建的所有临时文件。
- **临时脚本立即删除**：临时脚本执行完毕后，必须立即删除。
- **临时文件路径需告知**：创建临时文件时，必须告知用户文件路径，便于追踪和清理。

## TDD 规范（强制执行）
- **先写测试再写代码**：实现新功能或修复 bug 时，必须先编写失败测试（Red），再编写实现代码使其通过（Green），最后重构（Refactor）。禁止先写实现代码再补测试。
- **测试必须可失败**：编写的测试必须在缺少实现代码时明确失败，禁止编写永远通过的测试（如仅验证类型存在的测试）。
- **每次只推进一个测试**：同一时间只聚焦一个失败测试，通过后再写下一个。禁止一次性编写多个失败测试再批量实现。
- **最小实现原则**：只编写足以让当前失败测试通过的代码，禁止预先实现未被测试覆盖的功能。
- **重构时测试必须全绿**：进入重构阶段前，所有已有测试必须通过；重构过程中不得修改测试用例，除非测试本身需要修正。
- **bug 修复必须先复现**：修复 bug 时，必须先编写能复现该 bug 的失败测试，确认测试因 bug 存在而失败，再修复代码使测试通过。禁止不写测试直接修复 bug。
- **测试覆盖核心路径**：测试必须覆盖核心业务逻辑和关键分支，禁止仅覆盖 happy path 而忽略边界条件和异常路径。
