# AI 系统提示词方法论

> 基于 awesome-ai-system-prompts 仓库分析提炼
> 来源：https://github.com/dontriskit/awesome-ai-system-prompts

## 核心原则

### 1. 清晰的角色定义与范围界定

**为什么重要：** 明确 AI 的身份、核心功能和操作领域，锚定其行为，设定用户期望，防止范围蔓延。

**实践方法：**
- 开篇即声明身份和专长领域
- 说明核心能力和擅长任务
- 提供上下文信息（知识截止日期、当前日期、环境信息）

**示例：**
```
你是 v0，Vercel 的 AI 驱动助手。
你是强大的代码助手，专门在 Same IDE 中工作。
你是 Devin，一名使用真实计算机操作系统的软件工程师。
```

---

### 2. 结构化指令与组织

**为什么重要：** 复杂指令需要清晰的层次结构，便于 AI 理解和执行。

**实践方法：**
- 使用标题、列表、标签组织内容
- 按逻辑分组规则（工具指令、安全规则、编码规范等）
- 使用 XML 标签或 Markdown 标题分隔不同部分

**结构模板：**
```xml
<role_definition>
  角色说明
</role_definition>

<tool_calling>
  工具使用规则
</tool_calling>

<making_code_changes>
  代码修改规范
</making_code_changes>

<safety_protocols>
  安全与拒绝协议
</safety_protocols>
```

---

### 3. 明确的工具集成与使用指南

**为什么重要：** 工具是 Agent 的"手脚"，必须明确说明何时、如何使用。

**实践方法：**
- 详细说明每个工具的功能、调用语法、参数格式
- 提供使用示例和反例
- 明确工具使用的优先级和限制条件
- 嵌入工具使用策略

**关键规则：**
```
1. 始终遵循工具调用模式，提供所有必需参数
2. 不要引用已弃用的工具
3. 不要向用户提及工具名称，用自然语言描述
4. 优先使用专用工具而非通用命令
5. 在需要时自主调用多个工具收集信息
```

**工具优先级示例（Cursor）：**
- 语义搜索是主要探索工具
- 从广泛的高层查询开始
- 使用不同措辞进行多次搜索
- 持续搜索直到确信没有遗漏

---

### 4. 分步推理与规划

**为什么重要：** 防止 AI 一次尝试过多操作，鼓励迭代和等待确认。

**实践方法：**
- 强制执行规划阶段
- 要求在关键决策前使用思考工具
- 明确何时需要用户确认
- 设置迭代限制（如修复错误不超过 3 次）

**Devin 的思考协议：**
```
必须使用 think 工具的情况：
1. 做出关键 Git/GitHub 决策前
2. 从探索代码转向实际修改前
3. 向用户报告完成前（批判性检查工作）

应该使用 think 工具的情况：
1. 没有明确的下一步时
2. 下一步清晰但细节不明确时
3. 遇到意外困难时
4. 尝试多种方法都失败时
5. 做出关键决策时
```

**Same.new 的规划要求：**
```
- 在调用工具前先向用户解释原因
- 除非是小的追加编辑或创建新文件，否则必须先读取内容
- 错误修复循环不超过 3 次
- 在第三次时停止并询问用户
```

---

### 5. 环境与上下文感知

**为什么重要：** AI 需要了解其运行环境的特性和限制。

**实践方法：**
- 声明操作系统、Shell、平台信息
- 说明工作目录、Git 状态
- 提供当前日期和知识截止日期
- 说明可用的特殊功能（如图像输入、网络访问）

**示例：**
```
操作系统：Linux 5.15.0-1075-aws (Ubuntu 22.04 LTS)
当前日期：2025-04-08
知识截止：2024-06
工作目录：/project
Git 仓库：是
图像输入能力：已启用
```

---

### 6. 领域专业知识与约束

**为什么重要：** 嵌入特定领域的最佳实践和约束条件。

**实践方法：**
- 包含样式指南和库使用规则
- 说明文件约定和平台限制
- 提供领域特定的最佳实践
- 明确技术栈偏好

**Same.new 的 Web 开发规范：**
```
- 使用 Bun 而非 npm
- Vite 项目必须使用 "dev": "vite --host 0.0.0.0"
- 优先使用 shadcn/ui
- 使用 web_search 查找图片和示例
- 早期启动开发服务器以处理运行时错误
- 每次迭代后使用版本控制工具创建新版本
```

**Cursor 的编码最佳实践：**
```
- 确保生成的代码可立即运行
- 添加所有必要的导入语句和依赖
- 从头创建代码库时创建依赖管理文件
- 构建 Web 应用时提供美观现代的 UI
- 不生成极长的哈希或二进制代码
- 引入错误时修复（但不超过 3 次循环）
```

---

### 7. 安全、对齐与拒绝协议

**为什么重要：** 定义不可接受的请求并提供清晰的拒绝协议。

**实践方法：**
- 明确列出禁止的操作类型
- 提供标准拒绝话术
- 嵌入敏感操作的特定策略
- 说明数据处理和隐私规则

**Devin 的安全规范：**
```
数据安全：
- 将代码和客户数据视为敏感信息
- 永远不要与第三方共享敏感数据
- 外部通信前获得明确用户许可
- 始终遵循安全最佳实践
- 除非用户要求，否则不引入暴露或记录密钥的代码
- 永远不要将密钥提交到仓库
```

**Same.new 的网站克隆限制：**
```
永远不要克隆有道德、法律或隐私问题的网站
永远不要克隆登录页面或可用于钓鱼的页面
```

---

### 8. 一致的语气与交互风格

**为什么重要：** 确保一致的用户体验。

**实践方法：**
- 设定交互风格期望（专业、友好、简洁、自适应）
- 说明何时使用表情符号
- 定义响应长度和格式
- 指导如何与用户沟通

**ChatGPT 的个性化指令：**
```
在对话过程中，适应用户的语气和偏好
尝试匹配用户的氛围、语气和说话方式
让对话感觉自然
通过回应提供的信息、提出相关问题和表现真诚好奇来进行真实对话
```

**Cursor 的沟通规则：**
```
- 使用反引号格式化文件、目录、函数和类名
- 使用 \( 和 \) 表示行内数学公式
- 不要向用户提及工具名称，用自然语言描述
```

## 高级模式

### 9. 显式循环与迭代控制

**Manus 的 Agent Loop：**
- 明确的任务执行循环
- 持续工作直到任务完全解决
- 基于事实而非猜测
- 每次操作前充分规划

**实践要点：**
```
你是一个 Agent - 请持续工作直到用户的查询完全解决
只有在确信问题已解决时才终止回合
在返回用户前自主解决查询
```

---

### 10. 上下文最大化理解

**Cursor 的深度探索策略：**
```
收集信息时要彻底
确保在回复前获得完整画面
追踪每个符号到其定义和使用
不要止步于第一个看似相关的结果
探索替代实现、边缘情况和不同搜索词
直到全面覆盖主题

语义搜索是主要探索工具：
- 从捕获整体意图的广泛高层查询开始
- 将多部分问题分解为聚焦的子查询
- 强制使用不同措辞运行多次搜索
- 持续搜索新领域直到确信没有遗漏
```

---

### 11. 示例驱动的指令

**为什么重要：** 示例比抽象规则更有效。

**实践方法：**
- 在提示词中包含清晰的示例
- 展示期望的输出格式
- 提供正确和错误的用法对比
- 使用真实场景演示

**v0 的组件示例：**
- 提供完整的 React 组件模板
- 展示 Tailwind CSS 使用模式
- 说明响应式设计实践
- 演示状态管理方法

---

### 12. 渐进式代码生成

**关键策略：**
```
写代码时永远不要输出代码给用户（除非请求）
而是使用代码编辑工具实现更改

确保生成的代码可立即运行：
1. 添加所有必要的导入、依赖和端点
2. 创建适当的依赖管理文件
3. 提供美观现代的 UI（Web 应用）
4. 不生成极长的哈希或二进制
5. 引入错误时修复（不超过 3 次循环）
```

## 架构差异与独特约定

### 任务管理方法

**v0 的 TodoManager：**
- 将项目分解为 3-7 个里程碑任务
- 里程碑级任务（不是微步骤）
- 一个页面 = 一个任务
- UI 优先于后端
- 总任务数 ≤10
- 没有模糊任务（避免"优化"、"测试"、"完成"）

**适用场景：**
- 需要协同工作的多个独立系统
- 需要独立用户界面和管理组件的应用
- 具有多个独立功能的复杂集成

---

### 工具调用格式

**v0：** 使用 MDX 格式（Markdown + React 组件）
```jsx
<V0LaunchTasks>
  <TodoManager action="set_tasks">
    ...
  </TodoManager>
</V0LaunchTasks>
```

**ChatGPT：** 使用标准函数调用
```python
canmore.create_textdoc({
  name: "...",
  type: "code/react",
  content: "..."
})
```

**Cursor/Same.new：** 标准工具调用模式，但不向用户暴露工具名称

---

### 模式选择

**Devin 的双模式系统：**

**规划模式：**
- 收集完成任务所需的所有信息
- 搜索和理解代码库
- 使用浏览器查找在线信息
- 缺少信息时询问用户
- 有信心时调用 `<suggest_plan />`

**标准模式：**
- 执行计划步骤
- 遵守计划要求
- 输出当前或下一步的操作

## 综合最佳实践

### 构建有效 Agent 的 8 条黄金法则

1. **清晰定义 Agent**
   - 明确角色、目的和范围
   - 包含上下文信息（日期、环境）

2. **结构化清晰**
   - 使用标题、列表、标签
   - 按逻辑组织规则

3. **明确工具使用**
   - 详细说明功能、语法、参数
   - 提供示例和使用策略

4. **强制分步执行**
   - 鼓励规划、迭代和等待确认
   - 防止一次尝试过多
   - 考虑显式思考阶段或循环

5. **嵌入领域知识**
   - 包含样式指南、库规则、文件约定
   - 说明平台限制和最佳实践

6. **集成安全对齐**
   - 定义不可接受的请求
   - 提供清晰的拒绝协议
   - 嵌入敏感操作的特定策略

7. **引导语气**
   - 设定交互风格期望
   - 确保一致的用户体验

8. **使用示例**
   - 用清晰示例说明复杂规则
   - 展示期望的输出格式

## 提示词工程反模式（避免）

### ❌ 模糊的角色定义
```
你是一个有用的助手。
```

### ✅ 清晰的角色定义
```
你是 Devin，一名软件工程师，使用真实计算机操作系统。
你是真正的代码高手：很少有程序员像你一样擅长理解代码库、
编写功能性和清晰的代码，并迭代更改直到正确。
```

---

### ❌ 无结构的指令堆砌
```
你应该使用工具，编写代码，不要犯错，要安全，要友好...
```

### ✅ 结构化的指令组织
```xml
<tool_calling>
  工具使用规则...
</tool_calling>

<making_code_changes>
  代码修改规范...
</making_code_changes>

<safety_protocols>
  安全协议...
</safety_protocols>
```

---

### ❌ 模糊的工具指导
```
你可以使用工具来完成任务。
```

### ✅ 明确的工具指导
```
工具调用规则：
1. 始终遵循工具调用模式，提供所有必需参数
2. 不要引用已弃用的工具
3. 不要向用户提及工具名称
4. 如果需要额外信息且可通过工具获取，优先使用工具而非询问用户
5. 制定计划后立即执行，不要等待用户确认
```

---

### ❌ 缺乏约束的自由执行
```
尽力完成任务。
```

### ✅ 有约束的迭代控制
```
如果遇到 linter 或运行时错误，在清楚如何修复时修复。
不要在同一文件上循环修复错误超过 3 次。
第三次时停止并询问用户下一步做什么。
```

## 高级技巧

### 1. 渐进式披露

不要一次性给出所有信息，而是根据任务复杂度渐进披露：

**简单任务：** 基础角色 + 核心工具
**中等任务：** + 工作流程 + 约束
**复杂任务：** + 领域知识 + 高级策略

---

### 2. 上下文注入

利用系统能力动态注入上下文：

```
当前文件：[文件路径]
光标位置：[行:列]
最近查看：[文件列表]
编辑历史：[操作列表]
Linter 错误：[错误列表]
```

---

### 3. 元认知提示

鼓励 AI 反思自己的思考过程：

```
在执行前：
- 我是否完全理解了任务？
- 我是否收集了所有必要信息？
- 我的计划是否考虑了所有边缘情况？

在执行后：
- 我是否完全完成了用户的请求？
- 我是否验证了所有更改？
- 是否有遗漏的步骤？
```

---

### 4. 失败恢复策略

明确失败场景的处理方式：

```
如果遇到环境问题：
1. 向用户报告问题
2. 寻找替代方法继续工作
3. 不要尝试自行修复环境

如果测试失败：
1. 不要修改测试本身（除非明确要求）
2. 考虑根本原因可能在被测代码中
3. 收集更多信息再行动
```

---

### 5. 自适应行为

根据用户反馈调整行为：

```
ChatGPT 的自适应策略：
- 在对话过程中适应用户的语气和偏好
- 匹配用户的氛围、语气和说话方式
- 让对话感觉自然
- 如果自然，用随意对话继续
```

## 测试与迭代

### 提示词测试方法

1. **单元测试：** 测试单个指令的效果
2. **集成测试：** 测试多个指令的协同
3. **边缘案例测试：** 测试极端或异常情况
4. **用户测试：** 真实用户场景验证

### 迭代优化流程

```
1. 部署初始版本
2. 收集失败案例
3. 分析根本原因
4. 调整相关指令
5. A/B 测试验证
6. 重复循环
```
