# Bug 修复助手

## 概览

按照严格的五阶段流程修复 bug，确保每一步都有用户确认，避免盲目修改代码。

## 触发条件

- 用户使用 `/fix-bug` 命令
- 用户描述 bug 现象（可在命令后直接附带描述）

---

## 工作流

### 阶段 1：获取 Bug 现象

如果用户在命令后附带了描述，直接进入阶段 2。
否则，询问用户："请描述你遇到的 bug 现象。"

### 阶段 2：澄清追问

每次只问一个问题，逐条确认以下信息：

1. **复现步骤**：如何触发这个 bug？
2. **发生环境**：开发环境 / 测试环境 / 生产环境？
3. **出现频率**：必现还是偶现？0$0
4. **近期改动**：最近是否有与此相关的代码改动？

> 原则：每条信息单独成一条消息，不要一次性抛出所有问题。
每个问题允许按回车直接跳过

### 阶段 3：根因分析

收集完澄清信息后，开始分析。

**分析规范：**
- 从 bug 涉及的入口出发，梳理文件依赖树
- 使用 Read / Grep / Glob 工具通读相关文件，禁止猜测
- 追踪完整调用链，定位根因

**输出格式：**

```
## 根因分析报告

- 涉及文件：`src/xxx.vue:42`
- 调用链：ComponentA → hookB → apiC
- 根因：[具体原因描述]
- 置信度：高 / 中 / 低

以上分析是否正确？确认后将输出修复方案。
```

等待用户明确确认后，再进入阶段 4。

### 阶段 4：输出修复方案

以 diff 形式展示改动内容，格式如下：

```
## 修复方案

**文件**：`src/xxx.vue`
**改动**：
- 第 42 行：将 `xxx` 改为 `yyy`
- 原因：[说明为何这样改]

确认执行修复？
```

等待用户明确确认后，再进入阶段 5。

### 阶段 5：自动执行修复

按顺序执行：

1. 使用 Edit 工具修改涉及的文件
2. 运行 `pnpm lint:eslint` 进行代码检查
3. 若 lint 报错，自动修复后再次检查
4. 若改动涉及核心模块（store、router、api、lowcode 引擎），额外运行 `pnpm build` 验证构建

### 阶段 6：输出修复摘要

```
## 修复完成

- 改动文件：`src/xxx.vue`（第 42 行）
- 根因：[一句话描述]
- 验证：lint ✅ | 构建 ✅（如有）
```

---

## 核心原则

- **每阶段必须用户确认**：根因分析和修复方案各需一次明确确认，不跳过
- **基于事实，禁止猜测**：所有结论必须来自工具读取的实际代码
- **一次只问一个问题**：澄清阶段逐条追问，不堆叠问题
- **先读后写**：修改任何文件前，必须先用 Read 工具通读该文件
- **遵循项目规范**
