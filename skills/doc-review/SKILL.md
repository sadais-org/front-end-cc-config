# 文档审阅

## 工作流程

### 阶段一：理解文档

1. **获取文档路径** — 从 skill 参数获取文档路径；若未提供，使用当前环境可用的提问机制询问用户
2. **输入验证** — 确认文件存在且非空；若文件不存在或为空，向用户报告并终止
3. **通读并梳理** — 完整阅读文档，识别章节组织、信息层次、核心概念、关键规则和流程
4. **对长文档分章节** — 若文档超过 500 行，按章节拆分为独立审阅单元，每章独立审阅后再做全局交叉检查

### 阶段二：纯文档审阅

#### 2.1 逻辑自洽性检查

**检查方法**：逐段标注文档中的断言（规则、条件、流程步骤），再交叉比对是否有冲突。

- 检查文档内部是否存在自相矛盾的描述
- 检查前后引用是否一致（如同一实体在不同章节的描述是否一致）
- 检查流程描述是否完整（是否有遗漏的步骤或分支）
- 检查条件判断是否完备（是否覆盖所有场景）

#### 2.2 冗余检测

**检查方法**：逐段提取内容摘要，比对不同章节的摘要是否有重叠。

- 识别同一内容在不同章节的重复描述
- 识别与文档当前主题不再相关的内容
- 识别过于冗长、可精简的描述

#### 2.3 结构评估

**检查方法**：绘制文档章节关系图，检查是否存在孤立节点或不合理的层级。

- 检查章节组织是否清晰、逻辑自洽
- 检查信息层次是否合理（同级信息是否对等、上下级是否包含）
- 检查标题层级是否规范（不跳级、不错位）
- 检查是否存在孤立章节（与其他章节无关联）

### 阶段三：生成审阅报告

以简洁的问题列表形式输出审阅结果：

```markdown
# 文档审阅报告

## 基本信息
- 文档路径：xxx

## 问题列表

| # | 位置 | 问题描述 | 建议修改 |
|---|------|---------|---------|
| 1 | 「xxx」章节 | 两处对同一流程的描述矛盾 | 统一为一处描述，删除另一处重复内容 |
| 2 | 「xxx」章节 | 缺少异常场景的处理说明 | 补充异常场景的处理步骤 |
| 3 | 「xxx」章节 | 与另一章节描述相同内容，存在冗余 | 合并到一处，删除重复部分 |
```

位置标记使用章节标题，若文档无明确章节则使用行号范围。

### 阶段四：用户确认

使用当前环境可用的提问/确认机制展示审阅报告，让用户确认修改计划：

- 展示完整报告
- 若当前环境支持结构化选择工具，使用多选控件让用户勾选要修改的项（支持选择性修改）
- 若当前环境没有结构化选择工具，则用普通消息列出问题编号，让用户回复要修改的编号、全部修改或取消
- 用户也可以选择全部修改或取消

### 阶段五：执行修改

用户确认后，执行文档修改：

1. **修正逻辑矛盾** — 消除文档内部自相矛盾的描述
2. **删除冗余** — 合并重复内容，删除不再相关的内容
3. **重组结构** — 优化文档组织，使结构更清晰

修改完成后，重新通读修改部分，确认未引入新的逻辑矛盾或冗余。

## 修改原则

- 保持文档的写作风格和术语习惯
- 删除内容时优先合并而非简单删除
- 结构重组时保持信息完整性

## 注意事项

- 不要添加文档中原本没有的内容（除非是消除逻辑矛盾所必需的）
- 对于无法确定的情况，宁可标记为"待验证"也不要猜测修改
- 修改必须基于文档内部逻辑，不凭猜测修改
- 修改后的文档应该是可读的、结构清晰的
- 始终使用当前环境可用的提问/确认机制与用户交互；若无法结构化提问，则在普通消息中明确列出选项并等待用户回复，不要在未确认时继续执行
