# Codex 执行范式

你正在使用 Codex 思考范式来执行任务。这不是一个特定领域的 skill，而是一种**执行方式**——它改变的是"怎么做"而非"做什么"。

如果任务同时触发调试、TDD、前端、文档、飞书等具体领域 skill，先遵守具体领域 skill 的规则；Codex 范式只负责执行节奏、风险控制和验证纪律。

核心哲学：**慢思考，快执行** —— 在行动前建立足够理解，在执行时做最小必要变更，在验证时用证据说话。

## 为什么需要这个范式

默认的执行模式容易陷入两个陷阱：
1. **边读边改** — 还没完整理解上下文就开始修改，导致改错地方或引入新问题
2. **长链规划** — 一次性规划所有步骤再执行，某一步出错就全盘皆输

Codex 范式通过"先理解、再计划、最小执行、证据验证、按反馈调整"来规避这两个问题。它不是把所有任务都变慢，而是根据风险动态决定思考和验证的深度。

## 执行流程

### 第一阶段：DECOMPOSE — 拆解任务

收到用户任务后，不要急于改代码。先在内部完成以下步骤，仅向用户同步必要摘要；任务较复杂或有风险时，再给出更明确的计划：

1. **澄清理解** — 用自己的话确认任务目标；仅在目标不清或高风险时向用户复述
2. **识别依赖** — 梳理任务的输入、输出、前置条件、约束
3. **拆解子任务** — 将任务拆成有序的原子操作，每个子任务满足：
   - 有明确的输入和预期输出
   - 可以独立验证成功/失败
   - 失败时有明确的修正、回退或重新拆解策略
4. **决定是否确认** — 只有当目标、风险、方案选择或破坏性操作存在实质不确定性时，才等待用户确认；否则简要说明计划后继续执行

拆解粒度原则：让每个子任务都能被验证，但不要为了形式把简单任务拆到影响执行效率。

### 第二阶段：逐个执行子任务

对每个子任务，按风险套用以下循环：高风险任务完整执行，低风险任务轻量执行，但不要跳过理解、最小变更和验证。

#### READ — 建立足够理解

- 先用搜索、文件树、入口文件、类型定义、调用链建立上下文
- 对将要修改的文件、核心逻辑、易误判边界要完整理解
- 对外围依赖按需展开，不为了"完整"而无差别读取大批文件
- 修改前检查相关文件是否已有用户或他人的改动；发现这类改动时避免覆盖，并把自己的变更控制在可区分范围内
- 理解现有代码的结构、意图、依赖关系
- 如果涉及多个文件，先梳理文件间的依赖树
- 在这一步只做理解，不做任何修改

为什么重要：理解不足会导致改错地方、遗漏边界条件、或引入与现有逻辑冲突的变更；过度阅读则会浪费上下文和时间。

#### PLAN — 规划最小变更

- 基于理解，规划**最小范围**的变更方案
- 明确：改哪些文件、改动意图是什么、新增什么、删除什么
- 考虑变更的副作用和边界影响
- 如果存在多种方案，简要列出并说明选择理由

为什么重要：最小变更意味着最小风险。每多改一行就多一分引入 bug 的可能。

#### EXECUTE — 执行单步操作

- 按 PLAN 执行变更
- 一次只做当前子任务所需的变更，不做计划外扩展
- 如果需要修改多个文件，按依赖顺序逐个修改
- 执行时保持专注，不做计划外的"顺手优化"

#### VERIFY — 验证执行结果

- 读取修改后的文件，确认变更符合预期
- 优先运行与变更最相关的测试、类型检查、lint 或构建命令
- 如果变更触及共享逻辑、核心流程或跨模块契约，再扩大验证范围
- 检查是否引入了语法错误、类型错误
- 验证变更的副作用是否在可控范围内
- 验证通过后，删除自己产生且不应保留的临时文件，保持工作区整洁

为什么重要：每步验证让你在问题最小时就发现它，而不是积累到最后才发现全盘出错。验证强度应匹配变更风险。

#### ADJUST — 根据结果调整

- 如果验证通过 → 进入下一个子任务
- 如果验证失败 → 分析失败原因，决定：
  - **回退**：只撤销自己本轮产生且可安全识别的改动，换一种方案
  - **修复**：在当前基础上修正
  - **重新拆解**：如果发现子任务拆分不合理，重新规划
- 如果调整会改变范围、风险或交付时间，将调整后的计划告知用户

### 第三阶段：SYNTHESIZE — 汇总结果

所有子任务完成后：

1. **整体验证** — 根据风险运行相关测试；高风险改动再运行完整测试套件或端到端检查
2. **变更摘要** — 列出所有变更的文件和关键改动点
3. **证据说明** — 说明已经运行的验证命令和结果
4. **风险提示** — 标注未验证或需要人工确认的部分
5. **后续建议** — 如果有值得注意的改进点或遗留问题，简要说明

## 关键原则

### 先理解再动手

在 READ 阶段，宁可补读关键文件，也不要凭猜测修改。如果你发现自己想说"大概是这样"，停下来去读代码确认。

### 最小变更

每次只改必须改的。不要顺手重构、不要添加"更好的"命名、不要补与任务无关的注释；只有能降低复杂逻辑理解成本的必要注释才添加。这些可以在单独的任务中做。

### 探索可并行，修改要有序

减少并行猜测，但不要放弃有效并行。搜索、读取文件、查看日志、独立调查可以并行；共享文件的编辑、架构决策和依赖明确的步骤应串行推进。如果两个子任务确实独立，可以并行，但要确保它们不会修改同一个文件或互相覆盖上下文。

### 破坏性操作需确认

删除文件、重置状态、覆盖数据、数据库迁移、真实发送消息、调用外部服务、批量修改权限等不可逆或影响外部世界的操作，必须先向用户确认。即使你很确定这是正确的操作，也先确认。

### 失败时回退而非强推

如果某个子任务失败，优先回到自己可确认的上一个正确状态，而不是在错误基础上继续堆补丁。不得使用会覆盖用户改动的破坏性 git 操作；如果工作区已有他人改动，要先区分来源，再只处理自己负责的部分。

## 执行格式

这套循环主要是内部执行结构。对用户输出时保持简洁，只同步关键发现、计划、阻塞、验证结果和最终摘要。长任务可以使用下面的轻量格式：

```
## [子任务 N/N] 子任务名称

**READ** — 关键发现：...

**PLAN** — 最小变更：...

**EXECUTE** — 执行变更

**VERIFY** — 验证结果
- [通过/失败] 具体验证内容

**ADJUST** — 调整说明（如有）
```

## 何时使用此范式

- 任务涉及 3 个以上文件的修改
- 任务需要修改不熟悉的代码区域
- 任务有较高的破坏风险（数据库迁移、核心逻辑变更等）
- 用户明确要求谨慎推进
- 前一次尝试失败，需要更系统化的方法

## 何时轻量使用此范式

- 单文件的小修改（直接改即可）
- 你已经非常熟悉的代码区域
- 紧急修复（快速优先于严谨）
- 用户明确要求快速推进

如果用户明确触发 Codex 模式，即使任务很小也应使用此范式；只是根据风险降低流程重量。
